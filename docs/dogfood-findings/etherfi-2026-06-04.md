# EtherFi liquid-staking — Sluice dogfood / novel-bug hunt (2026-06-04)

Read-only review of `~/Data/etherfi-audit/etherfi/src` (non-corpus target). Goal: surface a
genuine unreported High/Critical and stress-test Sluice on a hardened modern protocol.

## Bottom line
**No confidently-exploitable High/Critical found.** The codebase is unusually hardened:
protocol-favoring ceiling division on withdrawals (`sharesForWithdrawalAmount`), post-burn
share-conservation asserts in the redemption manager, balance-snapshot post-conditions on every
priority-queue entry point, and checks-effects-interactions on all ETH-sending paths. Reported
honestly rather than inflating a Low.

## CONFIRMED FINDING (deep pass, 2026-06-04) — finalized-then-invalidated requests permanently leak into `ethAmountLockedForWithdrawal`
**Severity: Low (Medium ceiling if it accretes).** Not theft / not share-price — an over-lock that strands withdrawal liquidity, monotonically (no decrement path).

**Invariant:** `ethAmountLockedForWithdrawal == Σ getClaimableAmount(id)` over finalized + valid + unclaimed requests.

**Locations:**
- `WithdrawRequestNFT.invalidateRequest` — `WithdrawRequestNFT.sol:221-226` (flips `isValid=false`, no lock callback)
- `EtherFiAdmin._handleWithdrawals` — `EtherFiAdmin.sol:287-293` (adds `finalizedWithdrawalAmount` to the lock)
- `LiquidityPool.sol:233` — the ONLY decrement, reachable only via a successful `claimWithdraw`, which an invalidated request can never reach (`_claimWithdraw` requires `request.isValid`, `WithdrawRequestNFT.sol:145`).

**Walkthrough:** (1) Report N finalizes request R → `ethAmountLockedForWithdrawal += R_value`. (2) Report N+1 lists R in `withdrawalRequestsToInvalidate` → `invalidateRequest(R)` sets `isValid=false`. (3) R can no longer be claimed, so its `-= R_value` never runs. (4) Lock is permanently overstated by `getClaimableAmount(R)`. No function nets it out.

**Impact:** `EtherFiRedemptionManager.getInstantLiquidityAmount` (`EtherFiRedemptionManager.sol:248`) and the priority-queue gate (`LiquidityPool.sol:225`) subtract the inflated lock → instant redemptions + priority withdrawals throttled by the leaked amount, growing over time.

**Evidence it's a real asymmetry, not intended:** the priority-queue path already does the symmetric decrement — `PriorityWithdrawalQueue._cancelWithdrawRequest:549-555` decrements `ethAmountLockedForPriorityWithdrawal` when `wasFinalized`. The legacy NFT invalidation path is missing exactly that decrement.

**Fix:** make invalidation lock-aware — when invalidating a *finalized* request (`requestId <= lastFinalizedRequestId`), decrement the lock by its current `getClaimableAmount(requestId)`, mirroring the priority-queue cancel path.

**Why not High:** capped-claim math (`getClaimableAmount = min(amountOfEEth, amountForShare(shares)) - fee`) + the per-claim guard at `LiquidityPool.sol:228` mean aggregate claims can never *exceed* the lock — there is no under-lock / theft direction. Requires admin/oracle to invalidate *post*-finalization (unusual; invalidation is normally pre-finalization), so likelihood is low.

## Strongest residual lead (NOT confirmed — defense-in-depth, trusted path)
**`StakingManager.confirmAndFundBeaconValidators` (src/StakingManager.sol:167-192)** has no
idempotency / state-machine guard for the bulk (`validatorSizeWei - 1 ether`, ~31 ETH) deposit —
unlike `createBeaconValidators`, which enforces `REGISTERED → CONFIRMED` on `validatorCreationStatus`.
Its only guards are `etherFiNodeFromPubkeyHash(pubkeyHash) != 0` and a deposit-data-root match, so it
can be called repeatedly for the same pubkey. Why NOT a High: caller is gated by
`LIQUIDITY_POOL_VALIDATOR_APPROVER_ROLE` (trusted), and a duplicate top-up still goes to a validator
with ether.fi's own withdrawal credentials (recoverable on exit, not stolen). Real missing-invariant
worth a `CONFIRMED→FUNDED` consume guard for defense-in-depth; not attacker-driven fund loss.

## Triaged and rejected (with reasons)
1. `EtherFiRedemptionManager._calcRedemption` rounding — conserved (`sharesToBurn+feeShares == eEthShares`; asserts at L168/199 backstop drift).
2. `WithdrawRequestNFT._claimWithdraw` reentrancy (no `nonReentrant`, unbounded-gas `_sendFund`) — CEI correct: NFT burned + LP state mutated before the ETH send; same-id reentry reverts. Not exploitable.
3. `LiquidityPool.receive()` `totalValueOutOfLp -= msg.value` — worst case a revert (griefing a direct send); `getTotalPooledEther` preserved. No value leak.
4. Sluice F-001/F-002 (DepositAdapter "oracle manipulation") — FP: `stETH.balanceOf` reads measure actual transferred amount for stETH 1-2 wei rounding, not a valuation oracle.
5. Sluice F-010 (handleRemainder paired-variable drift) — FP: `totalRemainderEEthShares` / `aggregateSumOfEEthShare` used in disjoint phases; `handleRemainder` asserts `beforeEEthShares - eEthSharesMoved == eETH.shares(this)` (L282).
6. `PriorityWithdrawalQueue.cancelWithdraw` on finalized request — cancel/claim mutually exclusive via the same EnumerableSet; lock decremented symmetrically; rounding favors protocol.
7. `Liquifier.depositWithERC20` 1:1 stETH:eETH — documented de-peg design risk, cap/discount governed. Not an accounting bug.
8. MembershipManager V0→V1 `uint96` downcast (Sluice F-018) — one-time owner migration, not attacker-controlled at scale. Low.

## Best next-pass targets
- **`EtherFiOracle` consensus + `EtherFiAdmin._handleWithdrawals` → `ethAmountLockedForWithdrawal`
  solvency coupling.** The on-chain lock is driven entirely by oracle-reported
  `finalizedWithdrawalAmount` / `lastFinalizedWithdrawalRequestId`; the contracts assume these stay
  consistent with the actual sum of finalized-but-unclaimed NFT claim values. A mismatch (finalizing
  ids whose claimable sum ≠ reported locked amount, or invalidations not reflected in the lock) is the
  most plausible real solvency bug, but it lives across the off-chain oracle boundary — model the lock
  invariant end-to-end against a quorum-malicious / buggy oracle.
- **`EtherFiNode` / `EtherFiNodesManager` slashing & EigenLayer withdrawal accounting** — validator-
  lifecycle value flow back into `totalValueOutOfLp` (not fully traced this pass).

## Node / EigenLayer accounting deep pass (2026-06-04, wave 6) — 2 more confirmed Low defects + 1 latent risk
**Finding N-1 (Low, borderline Medium): `LiquidityPool.receive()` is permissionless and unconditionally rewrites the in/out TVL split.**
`LiquidityPool.sol:131-135` does `totalValueOutOfLp -= uint128(msg.value); totalValueInLp += uint128(msg.value);` with no caller restriction and no `fallback()`. (a) A direct ETH send (`v <= totalValueOutOfLp`) shifts out→in even though no out-of-LP ETH actually returned → permanent in/out-split desync feeding the liquidity gates at `:222/:225/:231` and every `min(balance,totalValueOutOfLp)` sweep; the donation itself is invisible to share price. (b) `v > totalValueOutOfLp` underflows/reverts → cheap repeatable griefing surface. Fix: restrict `receive()` to protocol senders + explicit `donate()` that credits `totalValueInLp`; at minimum clamp the decrement to `min(msg.value, totalValueOutOfLp)`.

**Finding N-2 (Low): `min(balance, totalValueOutOfLp)` sweep cap can strand returned validator ETH uncounted.**
Same pattern at `EtherFiNode.sol:131` (`completeQueuedETHWithdrawals`), `:162` (`sweepFunds`), `EtherFiRewardsRouter.sol:49`, `EtherFiRestaker.sol:155`, `Liquifier.sol:186`. When `contractBalance > totalValueOutOfLp` (driven low by concurrent sweeps / negative rebase / N-1 donations), a full-exit sweep forwards less than the real returned ETH; remainder stranded in the node until a later sweep. Not lost, but makes solvency ordering-dependent and is attacker-influenceable. Fix: track stranded balances, or apply the N-1 `receive()` clamp so forwarders can always send full balance.

**Latent risk (not currently exploitable — flag for the team): negative-slash rebase `uint128` wrap.**
`LiquidityPool.rebase` `:442`: `totalValueOutOfLp = uint128(int128(totalValueOutOfLp) + _accruedRewards)`. If a slash makes `|_accruedRewards| > totalValueOutOfLp`, the sum is negative and `uint128(neg)` wraps to a huge value → catastrophic TVL/share-price inflation. Currently gated ONLY by the off-chain `acceptableRebaseAprInBps` bound (`EtherFiAdmin.sol:257`), whose code comment says it "will be removed in future versions." Add an on-chain floor `require(int128(totalValueOutOfLp) + _accruedRewards >= 0)` before that guard is removed.

**Not confirmable on-chain (off-chain oracle trust):** EigenLayer reward double-count (oracle must exclude already-swept node ETH); these reduce to oracle correctness, not on-chain bugs.
