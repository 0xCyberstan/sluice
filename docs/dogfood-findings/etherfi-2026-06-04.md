# EtherFi liquid-staking — Sluice dogfood / novel-bug hunt (2026-06-04)

Read-only review of `~/Data/etherfi-audit/etherfi/src` (non-corpus target). Goal: surface a
genuine unreported High/Critical and stress-test Sluice on a hardened modern protocol.

## Bottom line
**No confidently-exploitable High/Critical found.** The codebase is unusually hardened:
protocol-favoring ceiling division on withdrawals (`sharesForWithdrawalAmount`), post-burn
share-conservation asserts in the redemption manager, balance-snapshot post-conditions on every
priority-queue entry point, and checks-effects-interactions on all ETH-sending paths. Reported
honestly rather than inflating a Low.

## Strongest residual lead (NOT confirmed — defense-in-depth, trusted path)
**`StakingManager.confirmAndFundBeaconValidators` (src/StakingManager.sol:167-192)** has no
idempotency / state-machine guard for the bulk (`validatorSizeWei - 1 ether`, ~31 ETH) deposit —
unlike `createBeaconValidators`, which enforces `REGISTERED → CONFIRMED` on `validatorCreationStatus`.
Its only guards are `etherFiNodeFromPubkeyHash(pubkeyHash) != 0` and a deposit-data-root match, so it
can be called repeatedly for the same pubkey. Why NOT a High: caller is gated by
`LIQUIDITY_POOL_VALIDATOR_APPROVER_ROLE` (trusted), and a duplicate top-up still goes to a validator
with ether.fi's own withdrawal credentials (recoverable on exit, not stolen). Real missing-invariant
worth a `CONFIRMED→FUNDED` consume guard for defense-in-depth; not attacker-driven fund loss.

## Triaged and rejected (with reasons)
1. `EtherFiRedemptionManager._calcRedemption` rounding — conserved (`sharesToBurn+feeShares == eEthShares`; asserts at L168/199 backstop drift).
2. `WithdrawRequestNFT._claimWithdraw` reentrancy (no `nonReentrant`, unbounded-gas `_sendFund`) — CEI correct: NFT burned + LP state mutated before the ETH send; same-id reentry reverts. Not exploitable.
3. `LiquidityPool.receive()` `totalValueOutOfLp -= msg.value` — worst case a revert (griefing a direct send); `getTotalPooledEther` preserved. No value leak.
4. Sluice F-001/F-002 (DepositAdapter "oracle manipulation") — FP: `stETH.balanceOf` reads measure actual transferred amount for stETH 1-2 wei rounding, not a valuation oracle.
5. Sluice F-010 (handleRemainder paired-variable drift) — FP: `totalRemainderEEthShares` / `aggregateSumOfEEthShare` used in disjoint phases; `handleRemainder` asserts `beforeEEthShares - eEthSharesMoved == eETH.shares(this)` (L282).
6. `PriorityWithdrawalQueue.cancelWithdraw` on finalized request — cancel/claim mutually exclusive via the same EnumerableSet; lock decremented symmetrically; rounding favors protocol.
7. `Liquifier.depositWithERC20` 1:1 stETH:eETH — documented de-peg design risk, cap/discount governed. Not an accounting bug.
8. MembershipManager V0→V1 `uint96` downcast (Sluice F-018) — one-time owner migration, not attacker-controlled at scale. Low.

## Best next-pass targets
- **`EtherFiOracle` consensus + `EtherFiAdmin._handleWithdrawals` → `ethAmountLockedForWithdrawal`
  solvency coupling.** The on-chain lock is driven entirely by oracle-reported
  `finalizedWithdrawalAmount` / `lastFinalizedWithdrawalRequestId`; the contracts assume these stay
  consistent with the actual sum of finalized-but-unclaimed NFT claim values. A mismatch (finalizing
  ids whose claimable sum ≠ reported locked amount, or invalidations not reflected in the lock) is the
  most plausible real solvency bug, but it lives across the off-chain oracle boundary — model the lock
  invariant end-to-end against a quorum-malicious / buggy oracle.
- **`EtherFiNode` / `EtherFiNodesManager` slashing & EigenLayer withdrawal accounting** — validator-
  lifecycle value flow back into `totalValueOutOfLp` (not fully traced this pass).
